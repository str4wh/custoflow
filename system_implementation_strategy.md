# System Implementation Strategy — CustoFlow

## 1. Implementation Schedule

- Phase 1: CustoFlow core platform
  CustoFlow’s core platform is implemented as a Flutter application with its entry point in `lib/main.dart`, where `Firebase.initializeApp(...)` connects the app to the `custoflow-39371` Firebase project and the named routes `/`, `/auth`, `/dashboard`, `/html-code`, and the fallback route handler are registered. The first functional layer is the landing experience in `lib/landinpage.dart`, which directs users into the onboarding flow through `navigateToAuth(bool isSignUp)`. The account lifecycle is handled in `lib/auth_page.dart`, where `AuthPage.handleSignUp()` creates a Firebase Auth account, generates an API key through `_generateApiKey()`, and writes the company record to `companies/{uid}` with the fields `id`, `name`, `adminEmail`, `apiKey`, `webhookUrl`, `createdAt`, and `departmentIds`. After sign-in through `handleLogin()`, the user is taken to `DashboardPage` in `lib/dashboard_page.dart`, which loads company data, creates departments through `DashboardPage._createDepartment(String name)` into `companies/{uid}/departments`, surfaces the API key, and generates the website integration snippet through `DashboardPage._generateIntegrationHtml()` and the `HtmlCodePage` screen in `lib/html_code_page.dart`.
- Phase 2: Webhook & feedback pipeline
  The webhook and feedback pipeline spans both account provisioning and downstream feedback ingestion. On the CustoFlow side, `AuthPage._sendToN8NWebhook(Map<String, dynamic> data)` posts a company creation payload to `http://localhost:5678/webhook-test/ddda45c7-ce0c-4340-9c36-7fe2f64ecd63` with `companyId`, `companyName`, `adminEmail`, `apiKey`, `webhookUrl`, `timestamp`, and `eventType: 'company_created'`. The same file also contains `_testWebhook()`, which probes the local n8n instance on port `5678`. For runtime feedback visibility, `DashboardPage._loadAnalytics()` reads `companies/{uid}/feedback`, counts the total documents, and aggregates `sentiment` values into the dashboard cards `Total Feedback`, `Positive`, and `Negative`. The repository’s `FIREBASE_SETUP.md` documents the intended Firestore rule structure for `companies/{userId}`, `departments/{departmentId}`, and `feedback/{feedbackId}`, which aligns with the company-scoped feedback storage model that the dashboard reads.
- Phase 3: M-Pesa clone integration for testing
  The testing integration couples the M-Pesa clone web application to CustoFlow’s feedback pipeline. In the supplied M-Pesa clone implementation, the feedback modules `js/feedback.js` and `js/public-feedback.js` post feedback data to CustoFlow’s n8n endpoint `http://localhost:5678/webhook-test/fd5bcb27-a283-45e1-ae9b-fba435fe24ba`. The generated CustoFlow website snippet from `DashboardPage._generateIntegrationHtml()` uses that same endpoint and sends `message`, `timestamp`, `companyId`, and `apiKey`, where `companyId` is derived from `_auth.currentUser?.uid` and `apiKey` is read from `_companyData?['apiKey']`. In the integrated test arrangement described for the M-Pesa clone, `js/feedback.js` and `js/public-feedback.js` include CustoFlow’s `companyId` and `apiKey` in their webhook payloads so that n8n can route submissions into the company-specific Firestore path `companies/{uid}/feedback`, which is the collection that the CustoFlow dashboard reads for analytics.
- Milestones achieved vs remaining
  Achieved milestones in the current CustoFlow codebase are the Flutter app shell, Firebase initialization, landing page, tabbed sign-up and login flow, password reset flow, company document creation in `companies/{uid}`, API key generation, onboarding dialog, department creation in `companies/{uid}/departments`, dashboard analytics reads from `companies/{uid}/feedback`, and integration snippet generation. Achieved integration milestones on the testing side are the documented M-Pesa clone feedback modules `js/feedback.js` and `js/public-feedback.js`, the use of the CustoFlow endpoint `http://localhost:5678/webhook-test/fd5bcb27-a283-45e1-ae9b-fba435fe24ba`, and the use of `companyId` and `apiKey` in webhook payloads for routing. Remaining milestones before production are to externalize Firebase and webhook credentials, replace `localhost` webhook targets with a hosted n8n URL, align the CustoFlow signup webhook endpoint with the feedback ingestion endpoint if they are intended to be the same workflow, operationalize Firestore and Storage security rules, and resolve the known M-Pesa clone issues around placeholder payment actions, route/section mismatches, and inconsistent user field naming.
- Estimated timeline for each phase
  As a planning estimate for reproducing and hardening the current implementation, Phase 1 is a roughly `2-3 week` effort because it includes the Flutter UI shell, Firebase Auth, Firestore company model, dashboard, department creation flow, and integration snippet generation. Phase 2 is a roughly `1-2 week` effort because it depends on n8n workflow setup, webhook connectivity validation, Firestore feedback ingestion, and dashboard analytics verification. Phase 3 is a roughly `3-5 day` effort when the M-Pesa clone feedback forms already exist, because the work is then focused on wiring `js/feedback.js` and `js/public-feedback.js` to the CustoFlow webhook, supplying the correct `companyId` and `apiKey`, and confirming that feedback lands in `companies/{uid}/feedback` and appears in the dashboard totals.

### Activities of Conversion

- Migrating manual customer feedback processes to CustoFlow automated pipeline
  The conversion objective is to move from manually handled feedback into a company-scoped, Firestore-backed feedback workflow. In CustoFlow, the administrative control plane already exists: company onboarding creates a `companies/{uid}` record, department setup writes to `companies/{uid}/departments`, and the dashboard reads `companies/{uid}/feedback`. The automated pipeline therefore centers on using the generated `apiKey`, `companyId`, and configured webhook URLs so that incoming feedback can be routed by n8n into the exact collection path the dashboard already understands.
- Connecting M-Pesa clone feedback submission forms to CustoFlow webhook for testing
  The testing conversion uses the M-Pesa clone’s existing feedback entry points instead of building a second test UI. Authenticated submissions originate from the `feedback-form` section handled by `js/feedback.js`, while anonymous submissions originate from the `public-feedback` section handled by `js/public-feedback.js`. In the integrated design you described, both modules POST to `http://localhost:5678/webhook-test/fd5bcb27-a283-45e1-ae9b-fba435fe24ba` and include CustoFlow’s `companyId` and `apiKey` so n8n can deposit the resulting feedback into `companies/{uid}/feedback` for CustoFlow analytics.
- Replacing hardcoded localhost webhook URLs with production endpoints before go-live
  Both codebases currently rely on hardcoded localhost webhook values. CustoFlow uses `http://localhost:5678/webhook-test/ddda45c7-ce0c-4340-9c36-7fe2f64ecd63` in `AuthPage._sendToN8NWebhook(...)` and `DashboardPage._createDepartment(...)`, while `DashboardPage._generateIntegrationHtml()` emits `http://localhost:5678/webhook-test/fd5bcb27-a283-45e1-ae9b-fba435fe24ba`. The M-Pesa clone testing modules also post to the `fd5bcb27-a283-45e1-ae9b-fba435fe24ba` endpoint. Before go-live, these values must be replaced with the hosted n8n URLs that correspond to the production workflows, and both codebases must be changed together so the payload contract remains aligned.
- Moving Firebase credentials from hardcoded values in lib/main.dart and js/firebase-config.js to secure environment configuration
  CustoFlow currently hardcodes its Firebase client configuration in `lib/main.dart`, and the M-Pesa clone currently hardcodes its Firebase client configuration in `js/firebase-config.js` according to the supplied documentation. For production, both should be externalized into environment-aware configuration so that development, staging, and production values are not embedded directly in source. This is particularly important because the same repositories also contain webhook targets and integration secrets such as the CustoFlow `apiKey` and `companyId` contract used by the M-Pesa clone feedback modules.

### System Conversion Strategy

- Parallel running: M-Pesa clone continues normal operation while CustoFlow webhook receives mirrored feedback data
  The safest conversion model for the current implementation is parallel running. The M-Pesa clone already has its own `feedback` collection behavior, feedback history view, and tracking workflow, so it can continue normal operation while `js/feedback.js` and `js/public-feedback.js` mirror each submission to CustoFlow’s n8n webhook. That allows the CustoFlow side to validate company routing, Firestore ingestion under `companies/{uid}/feedback`, and dashboard analytics without interrupting the M-Pesa clone’s native flows.
- Cutover plan: switch webhook URLs from localhost to production n8n instance
  Cutover should happen only after three validations pass: first, CustoFlow signup and department creation can reach the production n8n workflow; second, the M-Pesa clone authenticated and public feedback forms can POST successfully to the production endpoint; third, new documents appear in the correct `companies/{uid}/feedback` subcollection and are counted by `DashboardPage._loadAnalytics()`. Once those validations pass, the hardcoded `localhost:5678` values in `lib/auth_page.dart`, `lib/dashboard_page.dart`, `js/feedback.js`, and `js/public-feedback.js` can be replaced with the hosted n8n endpoint values.
- Rollback plan: revert webhook URL in feedback.js and public-feedback.js if issues arise
  Rollback on the testing integration is straightforward because the M-Pesa clone and CustoFlow are loosely coupled through the webhook URL. If production routing fails, the first rollback step is to revert the webhook targets in `js/feedback.js` and `js/public-feedback.js` so the M-Pesa clone stops forwarding to CustoFlow while its own application continues operating. If the failure also affects CustoFlow onboarding or department provisioning, the corresponding webhook URLs in `lib/auth_page.dart` and `lib/dashboard_page.dart` should also be reverted or disabled until the n8n workflow is corrected.

## 3. Training Plan

### Training Methods

#### CustoFlow Admin Training

- How to sign up and create a company account (AuthPage.handleSignUp, Firestore companies/{uid} write)
  Admin training should start with the registration path implemented in `AuthPage.handleSignUp()`. The trainee needs to understand that sign-up creates a Firebase Auth identity and then writes the company document to `companies/{uid}` with `id`, `name`, `adminEmail`, `apiKey`, `webhookUrl`, `createdAt`, and `departmentIds`. The admin should also be shown that the signup process asynchronously calls `_sendToN8NWebhook(...)` so welcome-email or provisioning automations can happen outside the main UI flow.
- How to create departments (DashboardPage.\_createDepartment, companies/{uid}/departments subcollection)
  Once signed in, admins should be trained to use the dashboard’s `Add Department` flow, which opens `_showCreateDepartmentDialog()` and persists the result through `DashboardPage._createDepartment(String name)`. Each created department is stored under `companies/{uid}/departments` with `name`, `createdAt`, and `webhookUrl`. Admins should understand that these records are the organizational units used to structure inbound feedback, even though the current Flutter app displays only the department name and webhook URL.
- How to copy and deploy the generated HTML integration snippet (DashboardPage.\_generateIntegrationHtml, HtmlCodePage)
  Training should cover the dashboard’s integration guide, which is triggered by `_showIntegrationGuide()` and built from `_generateIntegrationHtml()`. The generated snippet posts to `http://localhost:5678/webhook-test/fd5bcb27-a283-45e1-ae9b-fba435fe24ba` and includes `message`, `timestamp`, `companyId`, and `apiKey`. Admins need to understand that the snippet is a low-friction test integration artifact and that `HtmlCodePage` is the dedicated code display surface for copied HTML, even though the current dashboard flow primarily uses a dialog-based integration guide.
- How to view feedback analytics on the dashboard (sentiment counts from companies/{uid}/feedback)
  Dashboard training should focus on how `DashboardPage._loadAnalytics()` works, because that explains what data must exist for the cards to update. The dashboard counts the total documents in `companies/{uid}/feedback` and then increments the `Positive` and `Negative` cards based on the `sentiment` field stored on each feedback document. Admins should therefore be taught that if feedback is successfully ingested but does not contain a usable `sentiment`, it will affect total counts but not the positive or negative subtotals.
- How to use the API key for webhook authentication
  The `apiKey` is generated in `_generateApiKey()` and stored in `companies/{uid}`. Training should show where the API key appears in the onboarding dialog and the dashboard header card, and how to copy it from the UI. Admins should be told that the M-Pesa clone testing integration depends on passing this exact `apiKey` together with the correct `companyId` in the webhook payload, because that is the routing contract used to land feedback in the right `companies/{uid}/feedback` subtree.

#### M-Pesa Clone User Training (Integration Testing)

- How to submit authenticated feedback via feedback-form section (feedback.js)
  Authenticated testers should be trained to navigate to the `feedback-form` section in `index.html`, complete the form fields controlled by `js/feedback.js`, and submit the record while signed in. They should understand that this path is the logged-in test route for the CustoFlow integration and that it participates in both the M-Pesa clone’s native feedback handling and the webhook forwarding path used for CustoFlow testing.
- How to submit public feedback via public-feedback section (public-feedback.js)
  Public testers should use the `public-feedback` section, which is handled by `js/public-feedback.js` and does not require authentication. This training should emphasize that public feedback is still part of the integration test surface because it can post to the same CustoFlow webhook endpoint while exercising a different user journey and payload composition path.
- How to track feedback by reference number (track-feedback-modal)
  The M-Pesa clone includes `track-feedback-modal`, which looks up feedback by `referenceNumber`. Test users should be taught to preserve the reference they receive during submission and then use the tracking modal to confirm that the M-Pesa clone has accepted the feedback. This becomes especially useful when validating that a single submission can be observed in both the M-Pesa clone’s own tracking experience and CustoFlow’s company-scoped feedback analytics.
- How to view feedback history and filter by status, department, and search (feedback-history section)
  The `feedback-history` section lets logged-in testers review prior submissions and filter by `status`, `department`, and free-text search. Training here should focus on using the M-Pesa clone as the operational verification surface while CustoFlow is the analytical verification surface. In practice, a tester can submit through `feedback-form`, confirm visibility in `feedback-history`, and then verify that the same test run changed the CustoFlow dashboard totals under `companies/{uid}/feedback`.
- How to use the dashboard quick actions and balance toggle
  Although the payment actions are placeholders, testers still need to understand the surrounding navigation model. The M-Pesa clone dashboard contains the balance visibility toggle and quick actions for Send Money, Request Money, Pay Bill, Buy Airtime, and Withdraw, which are wired to `openAction(actionType)` for toast-style placeholder behavior. That orientation matters because feedback testing happens within the broader single-page app context and users must know how to reach the departments and feedback routes without confusing them with unfinished payment actions.

## 4. Software Maintenance Plan

- CustoFlow Firebase rules expiry: update rules expiry dates in Firestore console before they lapse
  CustoFlow does not currently version a `firestore.rules` file inside this repo; instead, `FIREBASE_SETUP.md` documents the intended Firestore rules for console entry. Operationally, that means the active ruleset must be tracked in Firebase Console and reviewed before any temporary or expiry-bound rule windows lapse. In the integrated environment, that review should be coordinated with the M-Pesa clone Firebase project because the testing pipeline depends on both the source application and the destination Firestore being usable at the same time.
- Firestore index management: monitor and create composite indexes as new queries are added
  The current CustoFlow Dart code only uses `.get()` reads on `companies/{uid}`, `companies/{uid}/departments`, and `companies/{uid}/feedback`, so no composite Firestore indexes are required by the visible Flutter code today. The M-Pesa clone, however, already versions `firestore.indexes.json`, and its feedback history and tracking features query root-level `feedback` data. As the combined system evolves, any new filtered or ordered queries added on the CustoFlow side, especially against `companies/{uid}/feedback`, should be monitored in Firestore error logs and formalized as indexes rather than left to ad hoc console creation.
- n8n webhook uptime: monitor localhost:5678 and migrate to hosted n8n instance for production
  The current integration depends entirely on a local n8n server at `localhost:5678`, which is acceptable for development but not for a production feedback pipeline. Maintenance should include connectivity checks for the `ddda45c7-ce0c-4340-9c36-7fe2f64ecd63` account-provisioning workflow and the `fd5bcb27-a283-45e1-ae9b-fba435fe24ba` feedback-ingestion workflow. The production roadmap should treat hosted n8n, or an equivalent always-on webhook service, as mandatory because the dashboard analytics and external web integration both depend on it.
- Flutter dependency updates: run flutter pub upgrade and test after major version bumps
  The dependency baseline is defined in `pubspec.yaml`, while the currently resolved versions are captured in `pubspec.lock`. Maintenance should use `flutter pub upgrade` deliberately rather than casually, because the app relies on Firebase packages and Flutter web behavior that can shift across releases. Every dependency update should be followed by `flutter analyze`, `flutter test`, and a manual web sign-up, login, department creation, and dashboard analytics validation pass.
- M-Pesa clone Firebase SDK version alignment: resolve mismatch between index.html (10.7.1) and login.html (9.22.0)
  The supplied M-Pesa clone implementation loads two different Firebase compat SDK versions across its two HTML entry points. That mismatch should be resolved as part of routine maintenance because authentication, Firestore, and storage behavior should not depend on which page initialized the session. Aligning the SDK version also reduces integration troubleshooting noise when validating webhook forwarding and feedback submission behavior.
- storage.rules currently denies all reads/writes (allow read, write: if false) — update before enabling file attachment features
  The M-Pesa clone’s current `storage.rules` state blocks all access, even though the documented features include `feedback-attachments/*` and `public-feedback-attachments/*`. Until that ruleset is changed, attachment functionality cannot be considered production-ready. This matters to the CustoFlow integration because an eventual webhook contract that includes attachment references will fail functionally if the originating app cannot persist or expose the underlying files.
- Fix known issues before production:
  `openAction()` payment buttons are placeholders, so quick actions do not represent real transactional functionality and must not be interpreted as production-complete wallet flows. `logout()` navigates to `'landing'` while the actual section id is `'landing-page'`, which is an implementation mismatch in the M-Pesa clone navigation layer. The protected routes `profile` and `transaction-history` have no matching section in `index.html`, creating dead-end route declarations. The `users` collection mixes `phone` and `phoneNumber`, which weakens data consistency. In CustoFlow, `provider`, `google_sign_in`, `shared_preferences`, and `fluttertoast` are declared in `pubspec.yaml` but unused in the current Dart sources, so dependency hygiene should also be cleaned up before a production hardening pass.

## 5. Change Management Plan

- Feature request and approval process for both CustoFlow and M-Pesa clone codebases
  Changes should be raised as explicit feature or defect requests that identify which codebase is affected, which files are in scope, and whether the change touches the shared webhook contract. On the CustoFlow side, requests often map directly to `lib/auth_page.dart`, `lib/dashboard_page.dart`, `lib/main.dart`, and the Firestore collections `companies/{uid}`, `companies/{uid}/departments`, and `companies/{uid}/feedback`. On the M-Pesa clone side, requests usually affect `js/feedback.js`, `js/public-feedback.js`, `js/navigation.js`, `js/auth-email.js`, `js/firebase-config.js`, `index.html`, or `login.html`. Approval should require explicit confirmation whenever a change alters `companyId`, `apiKey`, endpoint URLs, Firestore field names, or section/route identifiers.
- Version control strategy: branching model, commit conventions, pull request reviews
  Both repositories should use short-lived feature or fix branches rather than direct edits on the main line. Commit messages should identify the affected subsystem, for example `custoflow-auth`, `custoflow-dashboard`, `mpesa-feedback`, or `mpesa-navigation`, so webhook-contract changes are easy to find later. Pull requests should include at minimum a description of affected files, the exact route or collection names touched, the webhook URLs changed if any, and the manual test evidence for sign-up, feedback submission, and Firestore verification.
- Communication plan for deploying updates to company admins and M-Pesa clone users
  Changes that affect admins should be communicated in terms of visible UI and operational impact: sign-up flow changes, API key rotation, department creation behavior, dashboard analytics changes, or integration snippet changes. Changes that affect M-Pesa clone testers or users should call out whether the affected areas are `feedback-form`, `public-feedback`, `feedback-history`, `track-feedback-modal`, or dashboard navigation. Cross-system deployment notices must explicitly state if webhook endpoints or Firebase projects changed, because that impacts both the Flutter admin console and the web feedback source.
- Process for promoting a CustoFlow user to admin role
  The current CustoFlow implementation does not include a role field or role-management UI. In the present design, the company record owner created through `AuthPage.handleSignUp()` is effectively the admin because the Firebase Auth UID becomes the document id in `companies/{uid}` and the Firestore rules documented in `FIREBASE_SETUP.md` are based on `request.auth.uid == userId`. Any future admin-promotion process should therefore be treated as a schema change, not merely a UI change, because it would require new Firestore fields, updated rules, and corresponding dashboard authorization behavior.
- Process for adding new departments to CustoFlow and updating M-Pesa clone department dropdown
  Department additions begin in CustoFlow through `DashboardPage._createDepartment(String name)`, which persists department records in `companies/{uid}/departments`. If the M-Pesa clone department dropdown is intended to mirror CustoFlow departments, every new department must also be introduced into the M-Pesa clone’s selectable department source so both systems classify feedback consistently. That synchronization step should be part of the approved change checklist, because mismatched department names or identifiers will reduce the quality of downstream routing and reporting.
- Webhook URL change management: coordinating updates across both codebases when n8n endpoint changes
  Webhook URL changes are cross-repository changes and should never be made in only one place. CustoFlow currently embeds webhook URLs in `AuthPage._sendToN8NWebhook(...)`, `_testWebhook()`, `DashboardPage._createDepartment(...)`, and `_generateIntegrationHtml()`. The M-Pesa clone testing integration embeds the feedback-forwarding URL in `js/feedback.js` and `js/public-feedback.js`. A change request must therefore list every affected file and identify whether the change targets the account-provisioning webhook, the feedback-ingestion webhook, or both.
- Firebase security rules review schedule: quarterly review of Firestore and Storage rules for both projects
  Quarterly security reviews are appropriate because both systems depend on client-side Firebase access patterns that evolve as features are added. On the CustoFlow side, the review should verify that access to `companies/{uid}`, `companies/{uid}/departments`, and `companies/{uid}/feedback` still matches the intended user model. On the M-Pesa clone side, the review should cover `firestore.rules`, `firestore.indexes.json`, and `storage.rules`, with particular attention to the current all-deny storage state, the expiry-based Firestore rules model described in the supplied documentation, and any new attachment or feedback-tracking requirements introduced after testing.
