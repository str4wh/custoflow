# CustoFlow - Implementation Guide

## Overview

CustoFlow is a comprehensive customer feedback management system designed to automate the collection, processing, categorization, and analysis of customer feedback. The system consists of three integrated components working together seamlessly:

1. **Client Website Integration**: A lightweight JavaScript feedback form embedded in client websites (demonstrated with an M-Pesa clone application)
2. **N8N Automation Server**: A workflow automation pipeline that processes feedback, performs sentiment analysis, detects departments, sends notifications, and stores data
3. **CustoFlow Flutter Dashboard**: A cross-platform web/desktop/mobile application for viewing analytics, managing departments, generating reports, and accessing integration tools

The system eliminates manual feedback processing by automatically categorizing feedback as positive or negative, routing it to appropriate departments, notifying administrators via email, and providing real-time analytics through an intuitive dashboard interface.

## System Architecture

CustoFlow employs a three-layer microservices architecture:

### Layer 1: Client Website (M-Pesa Clone)
- **Technology**: HTML, CSS, JavaScript (Vanilla JS)
- **Component**: `public-feedback.js` - feedback collection form
- **Responsibility**: Capture user feedback (message and email) and submit to N8N webhook
- **Firebase Connection**: Stores raw feedback in M-Pesa Firebase (`feedback` collection)
- **Data Flow**: User interaction → Input validation → POST to N8N webhook → Local storage

### Layer 2: N8N Automation Server
- **Technology**: N8N (Node-based workflow automation)
- **Endpoint**: `http://localhost:5678/webhook-test/fd5bcb27-a283-45e1-ae9b-fba435fe24ba`
- **Responsibility**: 
  - Receive feedback via webhook
  - Perform AI-powered sentiment analysis (positive/negative detection)
  - Detect department category (Sales, Support, Technical, Billing, General)
  - Route feedback based on sentiment
  - Send email notifications to administrators
  - Store processed feedback in CustoFlow Firebase (separate collections for positive/negative)
- **Data Flow**: Webhook trigger → JavaScript analysis → Conditional routing → Email + Firestore → Response

### Layer 3: CustoFlow Flutter Dashboard
- **Technology**: Flutter 3.10.7+, Dart, Firebase (Auth + Firestore + Core)
- **Platform Support**: Web, Windows, macOS, Linux, Android, iOS
- **Responsibility**:
  - User authentication and account management
  - Real-time analytics visualization
  - Feedback message feed with filtering
  - Department management (create, view, copy webhook URLs)
  - Weekly HTML report generation and download
  - Integration code generation (HTML/JS with embedded credentials)
  - VS Code Copilot prompt generation
- **Data Flow**: Firebase Auth → Firestore queries → State management → UI rendering

## How Feedback Flows (Step by Step)

### Complete Feedback Journey

**Step 1: User Submits Feedback on Client Website**
- User clicks "Send Feedback" button on M-Pesa clone website
- Browser executes `public-feedback.js` script
- JavaScript prompts user for:
  - Feedback message (text input)
  - Email address (text input with validation)
- Email validation regex checks format: `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`
- If validation passes, continue to Step 2

**Step 2: Feedback Saved to M-Pesa Firebase**
- JavaScript prepares feedback object:
  ```javascript
  {
    message: "User's feedback text",
    email: "user@example.com",
    timestamp: "2026-03-28T14:30:00.000Z",
    sentiment: "pending",
    processed: false
  }
  ```
- Saves to M-Pesa Firebase Firestore: `feedback` collection
- Document ID auto-generated by Firestore

**Step 3: Webhook POST Sent to N8N**
- JavaScript executes `fetch()` POST request
- Target: `http://localhost:5678/webhook-test/fd5bcb27-a283-45e1-ae9b-fba435fe24ba`
- Headers: `Content-Type: application/json`
- Request body:
  ```json
  {
    "message": "User's feedback text",
    "email": "user@example.com",
    "adminEmail": "admin@company.com",
    "timestamp": "2026-03-28T14:30:00.000Z",
    "companyId": "yH3yxtik4wg9t1uH83cCJMKP2D02",
    "apiKey": "cfk_1769673888338_289906"
  }
  ```

**Step 4: N8N API Key Validation, Sentiment & Department Detection**
- **Webhook node** receives POST request and extracts JSON body
- **Code in JavaScript node** executes analysis function (includes API key validation, multilingual keywords):

    let sentiment;
    if      (hasNegative && !hasPositive)  sentiment = 'negative';
    else if (hasPositive && !hasNegative)  sentiment = 'positive';
    else if (hasNegative && hasPositive)   sentiment = 'negative'; // negative takes priority
    else                                   sentiment = 'positive'; // default

    // ── DEPARTMENT DETECTION (English + multilingual triggers) ──────────────
    const departmentKeywords = {

- **Rejected requests** (bad API key): routed away from email/Firestore nodes so invalid traffic is silently dropped

**Step 5: N8N Conditional Routing**
- **If node** evaluates condition: `{{ $json.sentiment }} == 'positive'`
- **True path**: Routes to positive feedback handlers
- **False path**: Routes to negative feedback handlers (higher priority notifications)

**Step 6: Gmail + SMS Notification Sent to Admin**
- **Send a message node** (for positive) or **Send a message1 node** (for negative)
- Gmail API integration configured with OAuth2
- **Optional SMS node** (Africa's Talking API) placed after the Gmail node on the negative branch:
  ```
  Node type: HTTP Request
  Method:    POST
  URL:       https://api.africastalking.com/version1/messaging
  Headers:
    apiKey:       <your Africa's Talking API key>
    Content-Type: application/x-www-form-urlencoded
  Body (form-urlencoded):
    username: <your AT username>
    to:       <admin phone e.g. +254712345678>
    message:  ⚠️ Negative feedback ({{ $json.department }}): {{ $json.message | truncate(100) }}
  ```
  - SMS fires **only on negative feedback** (after the IF node false branch)
  - For positive feedback the SMS node is bypassed; email is sufficient
- Email template:
  - **To**: `{{ $json.adminEmail }}`
  - **Subject**: 
    - Positive: "✅ New Positive Feedback Received"
    - Negative: "⚠️ URGENT: Negative Feedback Requires Attention"
  - **Body**: Includes message, customer email, department, timestamp
  - **Priority**: High for negative feedback

**Step 7: Firestore Document Creation**
- **Positive Path**: **Create a document node** → CustoFlow Firebase → `feedback` collection
- **Negative Path**: **Create a document1 node** → CustoFlow Firebase → `feedback negative` collection
- Document structure:
  ```json
  {
    "message": "User's feedback text",
    "email": "user@example.com",
    "adminEmail": "admin@company.com",
    "companyId": "yH3yxtik4wg9t1uH83cCJMKP2D02",
    "apiKey": "cfk_1769673888338_289906",
    "sentiment": "positive",
    "department": "Technical",
    "timestamp": "2026-03-28T14:30:00.000Z",
    "createdAt": Timestamp(2026-03-28 14:30:00),
    "status": "pending",
    "source": "website"
  }
  ```

**Step 8: N8N Webhook Response**
- **Respond to Webhook node** sends HTTP 200 OK
- Response body: `{ "success": true, "message": "Feedback received" }`
- Client JavaScript displays success message to user

**Step 9: CustoFlow Dashboard Updates (User-Initiated)**
- Admin logs into CustoFlow Flutter dashboard
- Dashboard fetches from Firestore:
  - `feedback` collection (positive feedback)
  - `feedback negative` collection (negative feedback)
- Filters by `companyId` matching admin's account
- Calculates analytics: total count, positive count, negative count
- Displays feedback in chronological feed with department filters
- Enables weekly report download with statistics

## Keyword Detection Logic

### Negative Sentiment Keywords
The system detects negative feedback using the following keyword list:

- **Quality Issues**: bad, terrible, awful, worst, horrible, poor, disappointing, subpar
- **Emotional Responses**: hate, frustrated, angry, annoyed, upset, disappointed, unhappy
- **Functional Problems**: broken, error, issue, problem, crash, bug, fault, defect, glitch
- **Performance**: slow, laggy, unresponsive, freeze, hanging, timeout
- **Experience**: difficult, complicated, confusing, unclear, hard, impossible
- **Resolution**: fix, repair, resolve, solve, help needed

**Detection Logic**: Message is flagged as negative if it contains ANY negative keyword and NO positive keywords. If both are present, negative takes priority.

### Positive Sentiment Keywords
The system detects positive feedback using the following keyword list:

- **Quality Praise**: good, great, excellent, amazing, outstanding, superb, brilliant
- **Emotional Responses**: love, like, enjoy, pleased, happy, satisfied, delighted
- **Superlatives**: best, perfect, wonderful, fantastic, awesome, incredible, phenomenal
- **Gratitude**: thank, thanks, appreciate, grateful, thankful
- **Recommendations**: recommend, impressed, blown away, exceeded expectations
- **Success**: works well, working perfectly, flawless, smooth, easy, simple

**Detection Logic**: Message is flagged as positive if it contains ANY positive keyword and NO negative keywords. Default to positive if no keywords match.

### Department Detection Keywords

#### Sales Department
**Triggers when message contains**:
- sales, sale
- purchase, buy, buying, bought
- pricing, price, cost, quote
- invoice, receipt, order
- product, products
- demo, demonstration
- trial, subscription

**Example Messages**:
- "I want to buy your premium plan"
- "Can I get a quote for enterprise pricing?"
- "Where is my invoice for last month?"

#### Support Department
**Triggers when message contains**:
- support, help, assist, assistance
- question, questions, asking
- how to, how do I, how can I
- need help, need assistance
- guide, tutorial, instructions
- account, login, password

**Example Messages**:
- "I need help with my account settings"
- "How do I reset my password?"
- "Can someone assist me with this feature?"

#### Technical Department
**Triggers when message contains**:
- technical, tech
- bug, bugs, error, errors
- crash, crashed, crashing
- not working, doesn't work, broken
- fix, fixing, repair
- code, API, integration
- server, database, connection

**Example Messages**:
- "The app crashes when I click Save"
- "Getting an error 500 on the API"
- "Technical issue with the integration"

#### Billing Department
**Triggers when message contains**:
- billing, bill
- payment, pay, paid
- charge, charged, charges
- refund, refunds
- subscription, subscribe
- card, credit card, payment method
- invoice, receipt, transaction

**Example Messages**:
- "I was charged twice this month"
- "Need a refund for my subscription"
- "Can't update my payment method"

#### General Department (Default)
**Assigned when**: No keywords match any specific department

**Common General Messages**:
- "Your company is amazing"
- "Just wanted to say hi"
- "Love your product"

## Firebase Collections

### CustoFlow Firebase (Company & Analytics Database)

#### Collection: `companies`
**Path**: `/companies/{userId}`

**Purpose**: Store company account information

**Document Structure**:
```json
{
  "id": "yH3yxtik4wg9t1uH83cCJMKP2D02",
  "companyId": "yH3yxtik4wg9t1uH83cCJMKP2D02",
  "name": "ABC Corporation",
  "adminEmail": "admin@abccorp.com",
  "apiKey": "cfk_1769673888338_289906",
  "createdAt": Timestamp(2026-01-30 10:15:00),
  "departmentIds": []
}
```

**Fields**:
- `id`: Firebase user UID (unique identifier)
- `companyId`: Same as id (redundant for compatibility)
- `name`: Company/organization name
- `adminEmail`: Administrator email for notifications
- `apiKey`: Unique API key (format: `cfk_[timestamp]_[random6digits]`)
- `createdAt`: Account creation timestamp
- `departmentIds`: Array of department document IDs (optional)

#### Subcollection: `companies/{userId}/departments`
**Purpose**: Store department configurations

**Document Structure**:
```json
{
  "name": "Customer Support",
  "createdAt": Timestamp(2026-01-30 10:20:00),
  "webhookUrl": "http://localhost:5678/webhook-test/ddda45c7-ce0c-4340-9c36-7fe2f64ecd63"
}
```

**Fields**:
- `name`: Department name (Sales, Support, Technical, etc.)
- `createdAt`: Department creation timestamp
- `webhookUrl`: N8N webhook endpoint for this department

#### Collection: `feedback` (Positive Feedback)
**Path**: `/feedback/{documentId}`

**Purpose**: Store all positive feedback responses

**Document Structure**:
```json
{
  "message": "Your product is amazing! Love the new features.",
  "email": "customer@example.com",
  "adminEmail": "admin@abccorp.com",
  "companyId": "yH3yxtik4wg9t1uH83cCJMKP2D02",
  "apiKey": "cfk_1769673888338_289906",
  "sentiment": "positive",
  "department": "General",
  "timestamp": "2026-03-28T14:30:00.000Z",
  "createdAt": Timestamp(2026-03-28 14:30:00),
  "status": "pending",
  "source": "website"
}
```

**Fields**:
- `message`: Customer feedback text
- `email`: Customer email address
- `adminEmail`: Admin to notify
- `companyId`: Company identifier (for filtering)
- `apiKey`: Authentication key
- `sentiment`: Always "positive" in this collection
- `department`: Detected department (Sales/Support/Technical/Billing/General)
- `timestamp`: ISO 8601 string from client
- `createdAt`: Firestore server timestamp
- `status`: Processing status (pending/reviewed/resolved)
- `source`: Origin of feedback (website/mobile/email)

#### Collection: `feedback negative` (Negative Feedback)
**Path**: `/feedback negative/{documentId}`

**Purpose**: Store all negative feedback responses (separate for priority handling)

**Document Structure**: Same as `feedback` collection, but `sentiment` field is "negative"

**Why Separate Collection**: 
- Easier to query urgent/negative feedback
- Different security rules for escalation
- Separate notification channels
- Performance optimization (smaller result sets)

### M-Pesa Firebase (Client Application Database)

#### Collection: `feedback`
**Path**: `/feedback/{documentId}`

**Purpose**: Store raw feedback from M-Pesa clone website before N8N processing

**Document Structure**:
```json
{
  "message": "The payment process is broken",
  "email": "user@example.com",
  "timestamp": "2026-03-28T14:30:00.000Z",
  "sentiment": "pending",
  "processed": false
}
```

**Fields**:
- `message`: Raw user feedback
- `email`: User email
- `timestamp`: Submission time
- `sentiment`: Initially "pending", updated after N8N processing
- `processed`: Boolean flag (false → true after N8N)

**Note**: This is the initial storage point. CustoFlow dashboard does NOT read from this collection.

## Firebase Security Rules

### CustoFlow Firebase Security Rules

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Company documents - users can only access their own
    match /companies/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
      
      // Departments subcollection
      match /departments/{departmentId} {
        allow read, write: if request.auth != null && request.auth.uid == userId;
      }
    }
    
    // Positive feedback collection
    match /feedback/{feedbackId} {
      // Admins can read their own company's feedback
      allow read: if request.auth != null && 
                     resource.data.companyId == request.auth.uid;
      
      // N8N can write feedback (authenticated via API key in application logic)
      // Allow unauthenticated writes for N8N webhook
      allow create: if true;
      
      // Admins can update status/notes on their feedback
      allow update: if request.auth != null && 
                       resource.data.companyId == request.auth.uid;
    }
    
    // Negative feedback collection - same rules as positive
    match /feedback negative/{feedbackId} {
      allow read: if request.auth != null && 
                     resource.data.companyId == request.auth.uid;
      allow create: if true;
      allow update: if request.auth != null && 
                       resource.data.companyId == request.auth.uid;
    }
    
    // Deny all other access
    match /{document=**} {
      allow read, write: if false;
    }
  }
}
```

**Security Notes**:
- Users can only read/write their own company data
- N8N writes allowed without auth (webhook integration)
- Feedback filtered by `companyId` in app logic
- Update permissions for status changes

### M-Pesa Firebase Security Rules

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Feedback collection - public write, authenticated read
    match /feedback/{feedbackId} {
      // Anyone can submit feedback (public form)
      allow create: if true;
      
      // Only authenticated M-Pesa admins can read
      allow read: if request.auth != null;
      
      // Only allow updating processed status
      allow update: if request.auth != null && 
                       request.resource.data.diff(resource.data).affectedKeys()
                       .hasOnly(['processed', 'sentiment']);
    }
    
    // All other collections require authentication
    match /{document=**} {
      allow read, write: if request.auth != null;
    }
  }
}
```

**Security Notes**:
- Feedback submission is public (no login required)
- Only admins can view submitted feedback
- N8N can update processed status
- Prevents unauthorized data access

## Known Issues & Workarounds

### Issue 1: Firestore Field Storage Problem

**Problem Description**:
N8N's Firestore node has a critical bug where it stores document fields as stringified JSON keys instead of proper Firestore field-value pairs. 

**Example of Incorrect Storage**:
Instead of storing:
```javascript
{
  message: "Great product",
  email: "user@test.com",
  companyId: "abc123"
}
```

N8N stores:
```javascript
{
  '{"message":"Great product","email":"user@test.com","companyId":"abc123"}': true
}
```

All data is encoded into a single JSON string used as the document key, with a boolean `true` value.

**Impact**:
- Firestore queries with `.where('companyId', isEqualTo: value)` return zero results
- Cannot filter by any field
- Cannot index fields for performance
- Data is essentially corrupted and unusable via standard Firestore queries

**Root Cause**:
- N8N Firestore integration node improperly serializes field mappings
- Likely a bug in how N8N handles the Firestore REST API
- May be related to expression evaluation or JSON stringification in N8N

**Flutter-Side Workaround Implemented**:

The Flutter dashboard includes advanced parsing logic to extract data from stringified keys:

```dart
// 1. Remove WHERE filter from Firestore queries
final positiveSnapshot = await _firestore
    .collection('feedback')
    .get();  // Fetches ALL documents

// 2. Filter in Dart code instead
final positiveMessages = positiveSnapshot.docs.where((doc) {
  // Check direct field access (normal Firestore)
  if (doc.data()['companyId']?.toString() == companyId) return true;
  
  // Check inside stringified keys (N8N bug workaround)
  for (final key in doc.data().keys) {
    if (key.contains(companyId ?? '')) return true;
  }
  return false;
}).map((doc) {
  // Combine all keys into searchable string
  final allKeysString = doc.data().keys.join(' ');
  
  // Helper to extract values from stringified JSON
  String extractValue(String fieldName) {
    // Try direct field access first
    final directValue = doc.data()[fieldName]?.toString();
    if (directValue != null && directValue.isNotEmpty) return directValue;
    
    // Parse from stringified key using regex
    final pattern = RegExp('"$fieldName"\\s*:\\s*"([^"]+)"');
    final match = pattern.firstMatch(allKeysString);
    return match?.group(1) ?? '';
  }

  return {
    'id': doc.id,
    'message': extractValue('message'),
    'email': extractValue('email'),
    'adminEmail': extractValue('adminEmail'),
    'companyId': companyId,
    'sentiment': extractValue('sentiment'),
    'department': extractValue('department'),
    'timestamp': extractValue('timestamp'),
    'createdAt': extractValue('createdAt'),
  };
}).toList();
```

**Workaround Components**:
1. **Fetch All Documents**: Remove Firestore WHERE filters, fetch entire collection
2. **Client-Side Filtering**: Filter by companyId in Dart using string matching
3. **RegEx Parsing**: Extract field values from JSON strings in document keys
4. **Fallback Logic**: Try direct field access first, then parse stringified keys
5. **Default Values**: Provide fallbacks for missing fields

**Debug Logging Added**:
```dart
// Debug code to inspect document structure
for (var doc in testSnapshot.docs) {
  debugPrint('=== Doc ${doc.id} ALL KEYS: ${doc.data().keys.toList()}');
  
  // Attempt to extract companyId from key strings
  String? docCompanyId = doc.data()['companyId']?.toString();
  if (docCompanyId == null) {
    for (final key in doc.data().keys) {
      if (key.contains('companyId')) {
        final match = RegExp(r'companyId["\s:]+([a-zA-Z0-9]+)').firstMatch(key);
        if (match != null) {
          docCompanyId = match.group(1);
          break;
        }
      }
    }
  }
  debugPrint('=== Extracted companyId: "$docCompanyId"');
}
```

**Performance Impact**:
- ❌ **Increased Latency**: Fetching all documents instead of filtered subset
- ❌ **Higher Bandwidth**: Downloading entire collection (not scalable)
- ❌ **Client Processing**: CPU usage for string parsing and regex matching
- ❌ **No Indexing**: Cannot use Firestore composite indexes
- ⚠️ **Scalability Concerns**: Will become problematic with 1000+ feedback documents

**Proper Solution Required**:
This workaround is temporary. The correct fix is to update N8N Firestore node configuration or use a custom webhook/Cloud Function to write to Firestore properly.

### Issue 2: OrderBy Clause Removed

**Problem**: Firestore `.orderBy()` clause causes query to fail when combined with `.where()` on fields that don't exist (due to stringified key issue).

**Solution**: Removed `.orderBy('createdAt', descending: true)` from Firestore queries. Sorting now happens in Dart using:
```dart
allMessages.sort((a, b) => _getFeedbackDate(b).compareTo(_getFeedbackDate(a)));
```

### Issue 3: Multiple Date Field Formats

**Problem**: Feedback documents have inconsistent date fields:
- Some use `timestamp` (ISO 8601 string)
- Some use `createdAt` (Firestore Timestamp or string)
- Some use milliseconds since epoch

**Solution**: `_getFeedbackDate()` method with fallback chain:
```dart
DateTime _getFeedbackDate(Map<String, dynamic> feedback) {
  final createdAt = _tryParseDate(feedback['createdAt']);
  if (createdAt != null) return createdAt;
  
  final timestamp = _tryParseDate(feedback['timestamp']);
  if (timestamp != null) return timestamp;
  
  return DateTime.fromMillisecondsSinceEpoch(0); // Fallback to epoch
}
```
